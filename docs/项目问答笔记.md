## 2. 项目定位:它解决什么问题

**一句话**:把 LangGraph agent 从"notebook demo"变成"能真正部署到生产 K8s 集群的服务",替你搞定 demo 跑通之后那 80% 跟 AI 无关、但上线必做的工程骨架。

它 **包含**:FastAPI + SSE 流式、鉴权/限流/输入校验、按次成本预算(超支 402)、多 LLM 供应商切换、可切换记忆后端、多版本灰度、结构化日志 + Prometheus + OTel、Docker/Helm/Terraform、CI 安全扫描、镜像签名 + SBOM。

它 **不包含**:OAuth2/OIDC、多租户身份、每租户 API key、计费。内置 `API_KEY` 只是单一共享密钥,面向内部/单租户。

**build-vs-buy 取舍**:适合想自己掌控基础设施/可观测/成本、不想被 LangGraph Platform 锁定、愿意自建的团队。想要托管控制台、不碰 K8s 的,直接用 LangGraph Platform 更省事。

## 3. 核心抽象:Domain Pack(插槽机制)

它不是"你写好 agent 套上去",而是**"规定一个写 agent 的标准姿势(契约),你按姿势写就自动获得全部生产能力"**。

**契约 `BaseDomainPack`(`pack_kernel/base_pack.py`)只强制实现三个方法**:
- `run(query)` — 同步执行返回结构化结果
- `arun(query)` — 异步执行
- `_iter_stream_events(query)` — 流式吐事件

再声明 `input_schema` / `output_schema`(Pydantic)。**pack 内部的 LangGraph 图爱多复杂多复杂,框架不管**,只要通过这三个口子交出去。

**自动白嫖的能力**:典型 HTTP 路由 `POST /packs/{id}/run` 和 `/run/stream`、请求体校验(不合法 422)、鉴权/限流/成本预算、会话历史、Prometheus 指标、多版本灰度、合规门禁(若在受监管目录)。

**最简例子** `hr_policy_qa`:只写了 schema(33 行)+ 一个 `build_prompt`(46 行),就得到上面全套能力。

## 4. 能否胜任复杂 Agent

**结论:完全胜任。** 你担心的多步/LangGraph/HITL/RAG/会话存储都有对应机制。

- **多步 + LangGraph 工作流**:`research_analysis` pack 是现成范例——在 `_build_graph` 里建标准 `StateGraph`,有节点、条件路由(`add_conditional_edges`)、共享 state(TypedDict)。想加循环/子图/并行随便加。
- **HITL 人机交互**:用 LangGraph 原生 `interrupt()` + `Command(resume=...)`,`examples/human_in_loop/graph.py` 有完整范例。**注意**:HITL 目前是 examples 独立脚本,没封装成 HTTP pack;"暂停态的 HTTP 编排"(第一次请求跑到中断、第二次带 resume 恢复,靠 checkpointer + 同 thread_id 续上)需要自己写。
- **RAG/检索**:`connectors/` 抽象层是一等公民,内置 `RagConnector` + 可插拔向量库(Chroma/pgvector)。
- **会话状态**:`core/memory.py` 分两层(checkpointer + run history),都支持 sqlite/redis/postgres 三后端切换。

**心智模型**:你写你的 StateGraph(框架不干涉)→ 收敛到 run/arun/stream 三方法 → 框架层自动加上 HTTP/鉴权/成本/可观测/合规。

## 5. 十大生产痛点总览

| # | 痛点(大白话) | 解决技术 |
|---|---|---|
| 1 | agent 只是脚本不是服务 | FastAPI + SSE 流式 |
| 2 | 谁都能调、乱调、恶意调 | Bearer 鉴权、限流、输入校验、SSRF、安全头 |
| 3 | LLM 按字收费,账单会爆 | CostTracker 成本追踪 + 按次预算(402) |
| 4 | 换供应商要重写代码 | LLM 工厂 + 统一抽象(一个环境变量切换)+ mock |
| 5 | 多副本记忆对不上 | 可切换存储后端(sqlite/redis/postgres) |
| 6 | 上新版怕搞崩线上 | 多版本注册 + 权重灰度 + 粘性会话 |

## 6. 痛点2:安全

绝大多数防护通过 **FastAPI 中间件**实现(请求进业务前依次穿过的安检门)。执行顺序:安全头 → 鉴权 → 限流 → 日志 → 关机拦截 → 请求体大小限制 → 业务。

### 限流(`core/security.py` RateLimiter)
- **算法**:滑动窗口。每个来源一个 `deque` 记时间戳,新请求先剔除窗口外旧记录,剩余数 ≥ 上限(默认 60/60s)则拒(429)。始终精确,无"整点清零"漏洞。
- **单机 vs 多副本**:内存版 `RateLimiter`;多副本用 `RedisRateLimiter`(Lua 脚本原子完成"清旧→数数→加新",sorted set 存)。
- **Fail-open**:Redis 挂了选择放行而非拒绝(限流是非关键功能,不该挡正常用户)。
- **代理感知**:只有直连可信代理时才信任 `X-Forwarded-For`,防伪造头绕过限流。

### 输入校验 + SSRF + Prompt 注入(三个不同层面)
- **请求体大小限制**:解析 JSON 前先卡,超 1MiB 返回 413,分块上传边读边数。
- **文本校验 `InputValidator`**:只做长度限制、拒绝空字节、压缩多余换行。**故意不用正则黑名单拦注入词**(徒劳且误伤)。
- **Prompt 注入正解 `wrap_untrusted_content`**:把不可信内容用分隔符包起来,告诉 LLM"这是数据不是指令"。只是一层防护,配合严格 schema(`extra="forbid"`)。
- **SSRF `validate_outbound_url`**:关键在于**把域名真正 DNS 解析成 IP 再判断**是否环回/私有/元数据地址,防 DNS rebinding。

### 安全响应头(`add_security_headers`)
每个响应盖上:`X-Content-Type-Options:nosniff`(防 MIME 嗅探)、`X-Frame-Options:DENY`(防点击劫持)、`Referrer-Policy`、`Cache-Control:no-store`、`CSP`、生产环境 `HSTS`、删 `server` 头。

**Bearer 校验细节**:用 `hmac.compare_digest` 恒定时间比较、循环不提前 break,防计时侧信道攻击。

**设计哲学**:能在业务外拦的绝不进业务;防不住的别假装能防;生产该严就严。

## 7. 痛点3:LLM 成本计费

**原理**:计费 = 内置单价表 × token 数。靠 LangChain callback 在调用前后拦截。

### 计费公式(`compute_call_cost`)
一次调用费用 = 普通输入 token × 输入价 + 读缓存 token × 缓存价(默认输入价 10%)+ 写缓存 token × 写缓存价(默认 125%)+ 输出 token × 输出价;批量 API 再 ×0.5。

### 单价来源
- `COST_PER_1K` 内置各模型单价表;`resolve_model_pricing` 多级容错匹配(精确名→去日期→Bedrock 片段→前缀别名)。
- 支持 `LLM_COST_TABLE_PATH` 用 JSON 覆盖/扩展,不改代码。

### token 来源
- 调用后从响应读**真实**用量(兼容各厂商字段名)。
- 调用前无真实值,用**字符数 ÷ 4** 粗估。

### 两道闸(CostTracker,LangChain callback)
- **调用前 `on_chat_model_start`**:按最坏情况预估,预计超预算直接不让调(事前刹车)。
- **调用后 `on_llm_end`**:真实花费累加(加锁,多 agent 并发),累计超预算抛 `BudgetExceededError`。

### 其他
- 整条流水线共享一个 CostTracker,预算卡"整次运行累计"。
- 生产遇未知模型 fail-fast(`UnknownModelPricingError`),防漏算。
- 成本写入 Prometheus(按模型/供应商)。
- `BudgetExceededError` → `AgentBudgetExceededError` → HTTP **402**。预算配 `PACK_DEFAULT_BUDGET_USD`。

## 8. 痛点5:会话记忆存储

**核心:单后端制**。`MEMORY_BACKEND` 一个开关选一种存储,选中的就是最终归宿,**没有 Redis→PG 二级流转**。

### 两层独立存储(`core/memory.py`)
| 层 | 存什么 | 工厂 |
|---|---|---|
| Checkpointer(短期) | LangGraph 图中间状态,用于断点续跑/HITL 恢复 | `create_checkpointer` |
| Run History(长期) | 每次运行输入输出审计 | `create_run_history` |

两工厂都读同一个 `memory_backend`,是两套独立存储。

### 后端矩阵
- `sqlite`:AsyncSqliteSaver + ConversationMemory(本地文件,仅单副本)
- `redis`:AsyncRedisSaver + RedisRunHistory(注意:短期用 RedisSaver,长期用 RedisRunHistory,同 Redis 不同 key 结构)
- `postgres`:AsyncPostgresSaver + PostgresRunHistory
- 兜底:MemorySaver + ConversationMemory

### 为什么 sqlite 只能单副本
连的是容器本地文件,各 pod 隔离。Redis/PG 连外部共享服务,多副本读同一份,记忆一致。

### 关键设计
- 工厂模式 + 依赖注入:pack 构造接收 checkpointer 参数。
- `RunHistoryStore` Protocol 契约:换后端上层调用不变(面向接口)。
- `get_pack_version_for_session`:支撑痛点6 的粘性会话。
- 分环境容错:开发降级到 MemorySaver,生产 fail-fast(RuntimeError)——防"悄悄降级埋雷"。
- async/sync 双路径(流式用异步 saver,同步 run 走线程池)。

> 想要"冷热分层(Redis 热 + PG 冷)"需自己把 `create_checkpointer` 和 `create_run_history` 拆成读不同配置项。

## 9. 延伸:真实生产的记忆架构

### Redis 持久化(AOF/RDB)
- **RDB**:定期把内存拍快照存 dump.rdb(恢复快,两次间可能丢)。
- **AOF**:记录每条写命令,重启重放(更不易丢,可配每秒 fsync)。
- 都写在 Redis 服务器本地磁盘(K8s 挂 PV,或用云托管 Redis)。Redis 开持久化后本身就是能落盘的数据库。

### 三种生产模式
- **模式 A(本项目)**:单后端,短期长期同库,最简单,90% 团队够用。
- **模式 B(经典冷热分离)✅**:短期 Redis(带 TTL,空闲超时自动清)+ 长期 PG(永久可查),应用**直接双写**,分别读。**多数正经生产选这个,不需要 Kafka**。
- **模式 C(write-behind)**:Redis + Kafka 异步刷 PG。只在写入洪峰(每秒上万)才需要,agent 场景基本用不上。

### 双写/读的要点
- **写顺序**:先同步写 PG(账本不丢)再写 Redis(加速)。PG 单条 insert 几毫秒,客服量级无需异步。
- **更省心变体**:PG 为唯一真相,Redis 只当可重建缓存(cache-aside),消除双写一致性问题。
- **一致性**:让 Redis 可丢可重建 + message_id 幂等,不一致就自愈。
- **读**:热上下文先读 Redis,未命中回落 PG 并回填;查历史直接读 PG。
- **TTL**:每次活动续期,空闲超时 Redis 自动删,天然实现"N 分钟无消息结束会话",无需定时器。

### 大型多租户 SaaS 要不要 Kafka
- 不是"多租户"决定,而是"吞吐/下游消费方数量"决定。
- **Kafka 核心价值 = 扇出(fan-out)**:一条消息一次生产,多个下游(PG/向量库/数仓/计费/训练/告警)各自订阅,彻底解耦。其次是削峰、故障隔离、可重放。
- Kafka 不替代 Redis/PG,是它们之间的"血管/日志"。
- **有 Kafka 后的写顺序**:
  - 架构 A(Kafka 即 WAL):同步写 Redis(即时读)+ 同步写 Kafka(持久化),PG 变异步下游。此时不再"先写 PG"。
  - 架构 B(PG 真相源 + 事务发件箱 outbox):先写 PG(+outbox 同事务)再中继到 Kafka,适合计费等强一致子域。
  - 大型 SaaS 常两者混用。
- 多租户按 tenant_id 分区做隔离/防噪声邻居。**没命中"多下游/高吞吐"前,PG + Redis 就够。**

## 10. 痛点6:金丝雀灰度发布

**原理**:同一个 pack 同时注册多版本(1.0/2.0),各挂权重,请求按权重抽签;新版先给小流量探雷,可随时调权重放量或回滚。

### 实现(`pack_kernel/registry.py`)
- `PackVersion`:version + pack_cls + weight。`_registry: dict[pack_id, list[PackVersion]]`(类级共享 + 加锁)。
- `get()` 三段式:显式指定版本→直接返回;单版本→直接返回;多版本→按权重抽签(全 0 权重报错)。
- **抽签 `_select_weighted_pack_version`**:
  - 无 affinity_key:`random.choices` 纯随机加权。
  - 有 affinity_key:对 key 做 SHA256 → 固定小数 → 落固定权重区间 → **同一 key 恒定同一版**(无状态、多副本一致)。
- `set_weights()`:运行时改权重,不重启即可灰度/回滚(权重是相对值)。

### 版本选择三层优先级(`api/router_factory.py`)——关于"粘性"的澄清
1. **`X-Pack-Version` 请求头**:人工强制指定(**是"覆盖"不是"粘性"**,最高优先级)。
2. **会话历史查询 `get_pack_version_for_session`**:真正的**持久化粘性**——查共享存储里该 session 上次用的版本,钉死不变(跨副本一致,绑定会话生命周期)。**主力机制**。
3. **affinity_key 哈希**(`_rate_limit_key`:多租户按 token、否则按 IP):无状态确定性兜底粘性。
- 响应头 `X-Pack-Version-Used` 是结果回执(供观测),不保障粘性。

> 灰度节奏:`{1.0:100,2.0:0}` → `{95,5}` → `{50,50}` → `{0,100}`;出事 `{2.0:0}` 秒回滚。配合 `X-Pack-Version-Used` 按版本切分错误率/延迟/成本。

## 11. 痛点7:可观测性

**可观测性三支柱**(`core/observability.py`):Metrics(数字/发现异常)、Logs(细节/根因)、Traces(时间线/定位哪步慢)。

### 结构化日志
- JSON 一行一条(python-json-logger),可被 ELK/Datadog 解析检索。
- `RequestIdFilter`:用 `contextvars` 把 request_id 注入每条日志,串联一次请求全链路(跨异步/线程池)。
- `SanitizingFilter`:复用 `sanitize_log_data` 自动脱敏 key/token/password,防密钥入日志。

### Prometheus 指标
- 三类型:Counter(累计,如 http_requests_total)、Gauge(瞬时,如 active_pipelines)、Histogram(分布,算 P99)。
- 针对 LLM 长耗时定制分桶(桶拉到 300s)。
- `metrics_path_label` 用路由模板而非具体 URL 做 label,防基数爆炸。
- `/metrics` 端点(make_asgi_app)供 Prometheus 抓取;`timed_node` 装饰器记节点耗时。

### OpenTelemetry 链路追踪
- **OTel 是不是类似阿里鹰眼?** 概念对应(都源自 Google Dapper),但:**鹰眼 = 具体产品**(≈ Jaeger/Zipkin/Datadog);**OTel = 厂商中立的开放采集标准**(只管按统一格式采集上报,后端自选)。选 OTel 是为零 vendor lock-in。
- 实现:OTLP 导出(默认 4317)+ BatchSpanProcessor + service.name。
- `trace_span("research_node", {...})` 埋点,span 按父子拼成火焰图。
- **优雅降级**:OTel/Prometheus 没装或没开则零开销 no-op,可选。

### 三支柱排障闭环
指标告警发现异常 → Jaeger 定位慢在哪个 span → 用 request_id 去 ELK 看日志根因。

### 关于 LLM 层可观测性(重要补充)
- 本项目在 LLM 层**只挂了 CostTracker callback**(只记 token/成本),**不记 prompt/completion 内容、推理链、工具调用**。这属于**基础设施可观测性**,打不开 LLM 黑盒。
- `langsmith` 只是 langchain 的间接依赖,**未主动接入**。
- **LLM 应用可观测性**需 Langfuse / LangSmith:
  - **LangSmith**:LangChain 官方、闭源、对本项目零配置(设环境变量即可)。
  - **Langfuse**:开源、可自托管、框架无关,更契合本项目"不锁定"气质。
- **两层互补**:基础设施层答"系统健康吗",LLM 层答"模型干了啥"。

### 两套系统怎么打通(避免来回切平台)
- **核心**:一个贯穿两边的关联 ID(trace_id/request_id)。
- **层次一(5 分钟)**:把 request_id 透传进 Langfuse 当 trace metadata,两边用同一 ID 搜。
- **层次二(推荐止步于此)**:再把 Langfuse trace URL 写回结构化日志 → 双向一键跳转。
- **层次三(单一视图)**:用 OTel 让基础设施 span 和 LLM span 共享 trace_id(OTel 上下文自动嵌套,加 OpenInference/OpenLLMetry 埋点),全送进一个支持 OTLP 的后端(Langfuse OTLP / Datadog 等)。配置最重。
- 建议:先做层次一二,痛点消失就停,别过度工程。

## 12. 痛点8:部署与阿里云对应

**从代码到生产 K8s 的五层桥**(`infra/`):

1. **Docker**(`infra/Dockerfile`):打包成到处能跑的镜像。多阶段构建(镜像小)、非 root 运行、健康检查。
2. **Docker Compose**:本地一键起 app + Redis + PG(单机编排)。
3. **Helm**:K8s 部署模板(带变量),dev/prod 各一套 values。已配齐 deployment/service/ingress/secret/hpa/keda/networkpolicy/pdb/servicemonitor。
4. **Terraform**:IaC 创建云上集群本身(GKE/EKS/AKS 三套入口)。层次:Terraform 盖楼→Helm 开店→Docker 是货品。
5. **KEDA 自动扩缩**:按业务指标 `active_pipelines`(而非 CPU)扩容——因 AI 应用多在等 LLM,CPU 很闲,用 CPU 判断会失灵。

### 上阿里云:基本都有托管产品对应
| 开源/通用 | 阿里云托管产品 |
|---|---|
| Kubernetes | ACK(容器服务 K8s 版) |
| 镜像仓库(GHCR) | ACR(容器镜像服务) |
| Prometheus | ARMS Prometheus |
| Grafana | 托管 Grafana(ARMS 内置) |
| ELK 日志 | SLS(日志服务)或托管 Elasticsearch |
| OTel 链路追踪 | ARMS 应用监控/链路追踪 |
| Terraform | terraform-provider-alicloud 或 ROS |
| KEDA | ack-keda 组件 / AHPA |
| 密钥管理 | KMS + 凭据管家 |
| Redis/PG | 云数据库 Redis 版 / RDS PostgreSQL |

- "托管"含义:控制面/存储/界面/高可用给你搭好,你只需接入(装 exporter、配采集、配密钥),不用自己运维这些系统。
- **本项目 Terraform 是 GKE/EKS/AKS,没有阿里云版**,需用 alicloud provider 或 ROS 自己写;**Helm Chart 基本可搬到 ACK**(Ingress 注解/StorageClass/镜像地址要调)。

## 13. 痛点9:合规门禁

**含义**:对简历筛选、合同审查、HR 问答、财务备忘等高危受监管业务,默认锁死,逼你完成合规程序并在技术上强制"免责声明 + 人工复核"。核心哲学 **fail-closed(默认关闭)**。

### 四层防线
1. **默认锁死 + 403**(`assert_regulated_pack_runtime_enabled`):pack 在 `REGULATED_PACK_IDS` 里且没开 `REGULATED_PACKS_ENABLED=true` → 抛错 → HTTP 403。顺序:先 schema 校验(422)再合规门禁(403)。
2. **COMPLIANCE.md 清单**:每个受监管 pack 自带,列明法规背景 + 模板提供了什么 + operator 还必须自己做什么。把"开开关"和"合规责任"绑定。
3. **服务端强制注入免责声明**(`apply_compliance_output`):在 LLM 输出**之后**用代码强行覆盖 `disclaimer` 和 `human_review_required`,**LLM 无法删弱**(不信任 LLM,确定性代码兜底)。
4. **强制人工复核队列**(`_create_review_best_effort` + review_store):标了 `human_review_required` 的输出自动排入待审队列,AI 只能辅助,终审归人。

### 设计哲学
纵深防御:默认不可用 + 责任显式化 + 不信任 LLM + human-in-the-loop 兜底。把"合规"变成技术上绕不过去的硬约束,而非靠自觉。类比:处方药完整管控链(锁柜→处方→警示标签→医生签字)。

## 14. 结合业务:国际物流场景应用

### 判断某场景要不要门禁
AI 输出**一旦出错且无人复核就使用**,会导致以下任一后果就该上门禁:
1. 违法/被罚/被诉 2. 安全事故 3. 重大财务损失 4. 对个人的自动化决策(隐私/歧视)

### 物流场景分级
- 🔴 **强烈建议门禁**:海关 HS Code 归类、制裁/出口管制筛查、危险品分类申报、合同/提单审查(用内置 `contract_reviewer`)、关税税费测算、保险理赔评估、客户信用授信、招聘筛选(用内置 `talent_screening`)。
- 🟡 **看情况**:贸易单证生成(草稿给人核对=轻量门禁;直接对外=门禁)、客户 PII 跨境(数据合规)。
- 🟢 **一般不用**:内部知识问答、邮件起草、资讯摘要、普通客服 FAQ、排线优化建议。
- 关务归类/制裁筛查/危险品是物流"头号门禁场景",AI 绝不能终审。自建 pack 后登记为受监管即复用整套门禁。跨境需按目标法域本地化免责文案(`PackPolicy.compliance_disclaimer`)。

### AI 客服:门禁 + 知识库(RAG)如何配合
**两者解决不同问题,要配合,不是一回事**:
- **知识库/RAG**:解决"AI 不知道准确/最新法规"(准确性),检索权威语料喂 AI、给出处。
- **门禁**:解决"答了也不能算终审"(责任),强制免责 + 人工复核 + 转人工。
- **光有知识库不够**:RAG 也会读错、法规会变、且责任问题 RAG 不管。高危问题必须人工兜底。

**AI 客服分层设计**:
```
客户提问
 → 分类:普通咨询 还是 涉及法规/关务/制裁?
    ├─ 普通(货在哪/怎么下单)→ 普通 pack + RAG,无门禁,直接答
    └─ 涉规(能否空运锂电池/是否受制裁/HS码关税)
        → 受监管 pack + RAG(权威语料+版本管理)+ 门禁 + 低置信度/敏感自动转人工
```

**现成模板**:`hr_policy_qa` 就是"RAG + 门禁"组合范例,其 output schema 同时含 `citations`(RAG 出处)、`confidence`(置信度)、`escalate_to_hr`(转人工)、`human_review_required` + `disclaimer`(门禁)。照它改成物流版 `trade_policy_qa`(政策语料换外贸法规、转 HR 改转合规官)即可。

## 15. 意图识别:项目现状与生产做法

### 项目现状
- **没有独立的意图识别模块/通用分类器**,也没有非 LLM 分类器(embedding 相似度、fastText、传统 NLU 都没有)。
- 最接近的是两处,且都是**用 LLM 做分类/路由**的示范:
  - `examples/supervisor/graph.py`:LLM-as-router,`supervisor_node` 每次调 LLM 判断 query 该给哪个专家(research/code/data),返回 JSON `{"next":...}`,配白名单兜底 + `add_conditional_edges` 分发。**是教学 demo,不是生产意图系统**。
  - `domain_packs/productivity/support_triage`:把"意图/类别分类"做成结构化输出的 pack(`category`/`priority`/`sentiment` + `confidence` + `escalate` 转人工)。
- 注意:版本路由(`X-Pack-Version`/affinity)是**部署层灰度**,与"用户意图"无关,别混淆。

### 为什么纯 LLM 做意图识别不适合生产
意图识别是**每条消息的必经第一跳**,纯 LLM 有三宗罪:
- **慢**:几百毫秒~几秒的固定延迟税,加在所有请求上。
- **贵**:每条消息都烧 token,信息量却很低(就分个类),杀鸡用牛刀。
- **不稳定**:同样的话可能分类漂移,还得靠 JSON 解析 + 白名单擦屁股。

### 生产标准做法:分级级联(cascade),从便宜到贵
绝大多数请求在便宜层解决,只有长尾才升级到 LLM:
```
query
 → [语义缓存] 命中? ──是──► 直接返回意图
 → [规则/正则] 命中高置信模式? ──是──► 直接返回(高危词强制走这)
 → [轻量分类模型] conf ≥ 阈值? ──是──► 采用
 → [LLM 兜底] 处理长尾 ──► 结果回写缓存 + 采样进训练集
```
外加:多意图/意图切换(多标签)、槽位填充(意图+实体一起抽)、拒识兜底意图(`unknown`)、badcase 回流闭环、逐层降级。

### 真实开源项目佐证(都不是纯 LLM 路由)
| 项目 | 做法 | 印证点 |
|---|---|---|
| **[Rasa](https://github.com/RasaHQ/rasa)** | pipeline:`RegexFeaturizer`(规则)→ `DIETClassifier`(轻量训练模型)→ `FallbackClassifier`(双阈值 `threshold`+`ambiguity_threshold` 拒识) | 规则 + 小模型 + 阈值拒识的工业标准 |
| **[semantic-router](https://github.com/aurelio-labs/semantic-router)** | embedding + 余弦相似度 + 阈值,写示例话术即用,微秒级,支持本地模型 | 官方明确"用向量空间替代又慢又不稳的 LLM 路由" |
| **[GPTCache](https://github.com/zilliztech/gptcache)** | 语义缓存(非精确匹配),命中直接返回 | 官方数据:成本降 10x、命中快 2-10x |

**总层次汇总**:语义缓存(GPTCache 类)→ 规则/正则(Rasa RegexFeaturizer)→ 轻量分类(DIET/semantic-router)→ 阈值拒识(FallbackClassifier)→ LLM 兜底(仅长尾)。

### 完整意图识别库清单
- **完整 NLU 框架(意图+实体一体,自带训练)**:Rasa(Python,工业标准,含对话管理)、[NLP.js/node-nlp](https://github.com/axa-group/nlp.js)(Node,40+语言,开箱即用)、[Snips NLU](https://github.com/snipsco/snips-nlu)(规则+统计双引擎,**已归档**只可参考)、spaCy `textcat`(+`EntityRuler`)。
- **路由/决策库(LLM 时代,轻量免训练)**:semantic-router、GPTCache、Cohere Classify / OpenAI few-shot。
- **云托管(免运维)**:微软 CLU(原 LUIS)、Google Dialogflow CX、AWS Lex、阿里云云小蜜/智能对话分析。

## 16. 意图识别应做成独立基础层(方案)

按项目"横切能力沉淀到 `core/`"的哲学,意图识别应与 `security`/`cost`/`memory` 平级,做成 `core/intent/`,在请求路由到具体 pack **之前**运行(分诊台)。

### 位置
```
请求 → [中间件:安全头/鉴权/限流]
     → [core/intent 意图识别层]  ← 新增基础层(分诊)
          缓存命中? → 规则命中? → 向量分类(带阈值) → LLM兜底
     → 路由决策:普通客服 pack / 受监管 pack + 合规门禁 / 转人工
     → pack.run()
```

### 最终拍板的方案:四级级联,最大化复用项目自有基建,不引入重依赖
```
query
 ├─① 语义缓存   命中(相似度≥0.95) ──► 直接返回          复用 core/vectorstore.py + Redis
 ├─② 规则解析   正则/词典命中 ──────► 直接返回(高危词强制) 新增 rules.yaml,零模型零成本
 ├─③ 向量分类   top≥0.75 且 与次高差≥0.1 ──► 采用        复用 core/vectorstore.py 做 KNN(双阈值抄 Rasa)
 ├─④ LLM 兜底   仅③不达标才调 ──────► 结构化枚举输出       复用 core/llm.py + CostTracker(超预算自动 402)
 └─ 全程低置信 ──► should_escalate=True ──► 转人工          复用 review_store
```
**关键取舍**:GPTCache/Rasa/semantic-router 的核心能力(缓存、向量分类)项目已有(`core/vectorstore.py` + embedding),为保持依赖精简**不引入**它们,只新增一个规则引擎 + 薄薄的级联编排。LLM 兜底结果回写缓存形成闭环,模型越用越少。

### 落地形态(对齐现有范式)
- 新增 `core/intent/`:`base.py`(`IntentClassifier` Protocol + `IntentResult{intent,confidence,source,slots,should_escalate}`)、`cascade.py`、`rules.py`、`vector.py`、`llm.py`、`intents.yaml`、`rules.yaml`。
- `create_intent_classifier(settings)` 工厂 + `INTENT_ENABLED`/`INTENT_BACKEND`/各层阈值(照抄 `create_checkpointer`)。
- 分环境优雅降级(向量/LLM 挂 → 降级规则层 → 兜底转人工)。
- 打 Prometheus(按 `intent`/`source` 看各层命中占比,重点盯"走到 LLM 的比例")+ request_id 日志,复用 `core/observability.py`。
- 规则与话术放 YAML,运营可改不动代码(同 `LLM_COST_TABLE_PATH` 思路)。
- 接入点:`api/router_factory.py` 分发前加 `resolve_intent()`,识别高危意图 → 路由受监管 pack → 自动触发合规门禁。

## 17. 拟提交的 PR:分层存储 + 意图识别

准备给上游提一个 PR(两块能力,必要时可拆两个),核心策略:**向后兼容 + opt-in 默认关闭 + 尽量复用现有基建、不塞重依赖**。

### PR 内容一:分层记忆(Redis 热 + 消息队列 + PG 冷)
- **现状**:`MEMORY_BACKEND` 单选,checkpointer 与 run history 跟同一开关(`core/memory.py`)。简单够用,不动它。
- **痛点**:单后端逃不开取舍——Redis 快但非可靠账本,PG 可靠但每次读写打库。
- **新增 opt-in 模式** `MEMORY_BACKEND=tiered`:Redis 存近期上下文(带 TTL 自动过期)+ PG 做可查询账本 + 消息队列(Kafka/Redpanda,小体量 RabbitMQ)异步扇出。
- **队列核心价值 = 解耦 + 扇出**(一条事件同时进 PG/向量库/计费,各订阅各的,还能扛写入突刺),不是"快"。
- **两种可配写入模式**:write-behind(同步写 Redis+发队列,消费者异步落 PG,适合高写入)/ outbox(同事务写 PG+outbox 再中继,适合计费等强一致)。读:先 Redis 未命中回落 PG 并回填。
- **对齐范式**:复用 `RunHistoryStore` Protocol、藏在现有工厂后、配置拆开让两层各指不同 tier、降级策略不变、延迟/不一致接入 observability。

### PR 内容二:意图识别基础层
即第 16 节的方案(四级级联 + 复用向量库/LLM/Redis/cost/observability + YAML 驱动 + 接合规门禁),默认 `INTENT_ENABLED=false`。

### 提交策略
- 先开 PR 聊设计方向再写实现;两块相对独立,可拆两个 PR。
- 留 Open questions(队列先支持哪个——倾向 Kafka;异步消费者跟主进程还是单独 worker;级联各层是否可单独开关),体现愿意讨论而非强推。

## 18. 还有哪些能力应沉淀为公共基础层

### 项目已有的(勿重复造轮子)
多供应商 LLM 工厂 + 超时 + 单次重试(`core/llm.py` + `BaseAgent` tenacity)、成本追踪与预算、记忆/checkpoint、工具框架(`core/tools.py`)、输出防护(`output_guard.py` 扫注入 + 可选二次 LLM 交叉核验)、eval 框架(确定性 checks + LLM judge + 数据集,见 `evals/`)、RAG connector、向量库、限流、可观测、合规门禁、灰度。

### 还缺、值得沉淀的公共能力(按优先级)

**第一梯队(最该补,明显缺失)**
1. **Prompt 管理与版本化**:现在 prompt 内联在各 pack 的 `build_prompt`。应抽成集中仓库 + 版本号 + 变量校验 + 按版本灰度(复用 `PackRegistry`)。
2. **LLM 响应缓存(语义缓存)**:项目只追踪成本不避免重复调用。加精确+语义两级缓存,复用 `core/vectorstore.py`。与意图识别的缓存是同一套基建,可合并。
3. **上下文窗口/token 预算管理 + 对话摘要**:`memory` 存全量历史,长对话会撑爆上下文。需滑动窗口 + 老对话自动摘要 + 按 token 裁剪。

**第二梯队(重要,场景相关)**
4. **韧性:模型降级链 + 熔断**:现在仅单次重试。应有主模型失败自动切备用模型/供应商 + 熔断短路。
5. **PII 检测与脱敏**:`output_guard` 扫注入、`sanitize_log_data` 按 key 脱敏日志,都非内容级 PII。跨境客户数据场景需独立 PII 层,并与合规门禁联动。
6. **出站 LLM 并发/配额限流(TPM/RPM)**:入站限流有了,出站没有,易打爆供应商配额。需令牌桶 + 并发信号量,与成本层同放。

**第三梯队(锦上添花)**
7. **RAG 质量层**:`rag_connector` 仅相似度检索,缺 rerank、混合检索(BM25+向量)、引用校验。
8. **反馈回收闭环**:采集点赞/点踩/纠正,回流 eval 数据集与微调(现在 eval 是离线静态集)。
9. **幂等键**:防重复提交重复跑昂贵 agent。

### 最该先做的三个
**Prompt 管理(1)、响应缓存(2)、上下文管理(3)**——每个 agent 必用、现在没有、且能与意图识别/存储 PR 共享同一批基建(vectorstore、registry 灰度)。1、2 可顺着分层存储 PR 的向量库改造一起做。
